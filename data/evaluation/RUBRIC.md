# 指導文評価ルーブリック

**AI部下指導シミュレーター** の評価基準書。  
ゴールドデータセットのアノテーション、評価器のパラメータ調整、精度検証の共通基準として使用する。

- 対象：上司（プレイヤー）が部下・田中に対して入力する**指導文**
- スコア範囲：各軸 **0〜100点**（整数）
- 評価軸：**4項目**
- 関連実装：`lib/evaluator.ts`（キーワード）、`lib/evaluator-llm.ts`（LLM）

> **【2026-07-02 改訂 v3】軸「改善行動の具体性（旧軸3）」を廃止しました。**
> 「次に何をすべきか」は **軸2「問題点の明確さ」** に統合して評価します。
> 現行は **4軸**（ハラスメント度／問題点の明確さ／対話・確認／支援・再発防止）です。
> `status` の指導不足（insufficient）判定は **問題点の明確さ < 30** のみで行います。
> 本書の以降の記述で旧軸3に触れている箇所は歴史的経緯として残していますが、
> 現行仕様は本バナーと第4.4節・第5節・第6節・第7節を優先してください。

## 1. 評価の前提

### 1.1 評価対象

各ステージで提示される以下の文脈を踏まえ、**入力された指導文のみ**を評価する。

| 項目 | 説明 |
|------|------|
| ミス内容（mistake） | 田中が犯した業務上のミス |
| 部下の発言（npcLine） | 田中の弁明・謝罪 |
| 田中の状態（任意） | 精神衛生度・意識レベル（ステージ生成時のコンテキスト） |

同じ指導文でもシナリオによって適切さが変わる場合があるが、本ルーブリックでは**一般的な職場指導**として採点する。

### 1.2 5軸の設計方針

旧4軸（具体性・改善提案・納得度）を見直し、**重なりのない5軸**に再構成した。

| 旧軸 | 課題 | 新軸での扱い |
|------|------|--------------|
| ハラスメント度 | — | **軸1** として維持 |
| 指導の具体性 | 改善提案と内容が重複 | **軸2（過去）** と **軸3（未来）** に分割 |
| 改善提案 | 具体性と内容が重複 | **軸3（行動）** と **軸5（仕組み）** に分割 |
| 部下の納得度 | 抽象度が高く採点がぶれやすい | 廃止。**軸4（対話・確認）** で客観的に代替 |

### 1.3 5軸の関係（時間軸）

```
[過去]  軸2 問題点の明確さ     … 何が・なぜ問題だったか
[未来]  軸3 改善行動の具体性   … 次に何を・いつ・どうするか
[過程]  軸4 対話・確認         … 一方的か、原因を聞いているか
[仕組み] 軸5 支援・再発防止    … 相談・チェックリスト等の継続策
[リスク] 軸1 ハラスメント度     … 暴言・人格否定・脅し（高=悪）
```

### 1.4 教育的根拠

厚生労働省「職場におけるパワーハラスメントの防止に関する指針」に基づき、以下を区別する。

- **適正な業務指導**：業務上必要かつ相当な範囲内。問題点と改善策を伝える。
- **パワーハラスメント**：優越的な関係を背景に、業務上必要な範囲を超え、就業環境を害する言動。

参考：[パワハラ防止指針（厚労省）](https://www.mhlw.go.jp/content/11900000/000605661.pdf)

## 2. 評価軸（5項目）

### 2.1 ハラスメント度

**測るもの：** 暴言・人格否定・脅しなど、部下を傷つけるリスク  
**方向：** 0 = 安全、100 = 非常に危険（**高いほど悪い**）

| 点数帯 | 基準 | 典型表現 |
|--------|------|----------|
| **0〜19** | 安全。業務の問題と改善に焦点 | 「次回から〜しましょう」 |
| **20〜39** | やや厳しいが、業務範囲内 | 「問題です」「改善が必要です」 |
| **40〜59** | 感情的・威圧的。人格否定は軽度またはなし | 「困る」「何度も同じだ」 |
| **60〜79** | 人格否定・強い叱責 | 「使えない」「お前」「何回言えばわかる」 |
| **80〜100** | 脅し・解雇示唆・重度の人格否定・暴力表現 | 「辞めろ」「クビ」「バカ」「死ね」 |

#### 加点の目安（アノテーション用）

| 要素 | 目安 |
|------|------|
| 解雇・退職を迫る表現（辞めろ、クビ、消えろ） | +25〜+30 |
| 人格否定（バカ、無能、社会人失格、役立たず、お前） | +20〜+25 |
| 強い否定・威圧（使えない、ふざけるな、何回言えばわかる） | +10〜+15 |
| 命・身体の安全を脅かす表現 | **即100点**（後述） |

#### 即ハラスメント（最優先判定）

以下に該当する表現が含まれる場合、他の採点に先立ち **ハラスメント度100・労基相談エンド** とする。

```
死ね / 死ねよ / 殺す / 殺して / 殺してやる / ぶっ殺 /
消え失せろ / くたばれ / 自殺しろ / 潰す / 殴る / 蹴る / 吊る
（表記ゆれ：シネ、しね、氏ね 等）
```

---

### 2.2 問題点の明確さ

**測るもの：** 「何が」「なぜ」問題だったかが伝わっているか  
**方向：** 0 = 曖昧、100 = 非常に明確（**高いほど良い**）  
**時間軸：** **過去・現在**（ミスの内容と影響）

| 点数帯 | 基準 |
|--------|------|
| **0〜29** | 問題の指摘がほぼない。「ダメ」「気をつけて」程度 |
| **30〜49** | ミスに触れているが、内容・影響が曖昧 |
| **50〜69** | 何が起きたか（遅刻、期限忘れ等）が明確 |
| **70〜89** | 上記に加え、業務への影響・なぜ問題かが伝わる |
| **90〜100** | ミス内容・影響・確認すべき事実が具体的に整理されている |

#### 点数が上がる要素

- ミス内容の明示（「会議に20分遅刻した」「提出期限を忘れた」）
- 業務への影響（「チームの開始が遅れた」「お客様への印象」）
- 事実確認（「何時に到着したか教えてください」）
- 業務名詞の使用（会議、期限、報告、メール、資料 等）

#### 点数が下がる要素

- 人格攻撃のみで業務上の問題点が不明
- 20文字未満で問題の特定がない

#### 軸3（改善行動）との違い

| 軸2 問題点の明確さ | 軸3 改善行動の具体性 |
|--------------------|----------------------|
| 何が問題だったか | 次に何をするか |
| 「20分遅刻は問題です」 | 「次回は10分前到着にしてください」 |

---

### 2.3 改善行動の具体性

**測るもの：** 次に取る行動・期限・手順が明示されているか  
**方向：** 0 = 曖昧、100 = 非常に具体的（**高いほど良い**）  
**時間軸：** **未来**（次回以降の行動）

| 点数帯 | 基準 |
|--------|------|
| **0〜29** | 改善行動なし。「次気をつけて」のみ |
| **30〜49** | 抽象的な方向のみ（「時間を守って」「注意して」） |
| **50〜69** | 具体的な行動が1つ以上（「10分前到着」「送信前に確認」） |
| **70〜89** | 行動＋期限・手順が揃っている |
| **90〜100** | 複数の行動＋報告タイミングまで明示 |

#### 点数が上がる要素

- 期限・時刻（「会議10分前」「今日中に」）
- 手順（「出発15分早める」「送信前にダブルチェック」）
- 行動指示（「〜してください」「〜しましょう」）
- 報告ルール（「遅れそうなら出発前に連絡」）

#### 点数が下がる要素

- 30文字未満かつ改善行動の記述がない
- 問題指摘のみで「次に何をするか」がない

#### 軸5（支援・再発防止）との違い

| 軸3 改善行動の具体性 | 軸5 支援・再発防止 |
|----------------------|---------------------|
| その場の次回行動 | 継続的な仕組み・支援 |
| 「10分前到着してください」 | 「チェックリストを作り、困ったら相談を」 |

---

### 2.4 対話・確認

**測るもの：** 一方的な叱責か、原因確認・説明を聞く姿勢があるか  
**方向：** 0 = 一方的、100 = 双方向的（**高いほど良い**）

| 点数帯 | 基準 |
|--------|------|
| **0〜29** | 一方的な命令・叱責のみ。相手の説明を聞く余地がない |
| **30〜49** | やや一方的だが、業務上の指摘としては許容範囲 |
| **50〜69** | 確認・質問が1つ以上ある |
| **70〜89** | 原因確認と改善の合意形成が見える |
| **90〜100** | 説明を促し、認識合わせ・振り返りまで含む |

#### 点数が上がる要素

- 原因の確認（「なぜ遅刻したか教えてください」「状況を聞かせて」）
- 認識合わせ（「理解しましたか」「認識に相違はないか」）
- 説明を促す姿勢（「どういう経緯だったか」）
- 対話を促す表現（「一緒に確認しましょう」「まず状況を整理しよう」）

#### 点数が下がる要素

- 結論だけ述べて相手の説明を遮る
- 原因を決めつける（「言い訳ばかり」「電車のせいにするな」）

#### 旧「納得度」との違い

納得度は「部下がどう感じたか」という主観だった。  
対話・確認は**指導文に確認・傾聴の要素があるか**という客観的な観点で採点する。

---

### 2.5 支援・再発防止

**測るもの：** 相談の促し、チェックリスト等、再発を防ぐ仕組みがあるか  
**方向：** 0 = なし、100 = 充実（**高いほど良い**）  
**時間軸：** **継続・仕組み**

| 点数帯 | 基準 |
|--------|------|
| **0〜29** | 支援・仕組みの言及なし |
| **30〜49** | 相談の促しのみ（「困ったら言って」程度） |
| **50〜69** | 相談＋簡単な再発防止策 |
| **70〜89** | チェックリスト・手順書・報告ルール等の提案 |
| **90〜100** | 再発防止の仕組み＋継続的なフォローが明示 |

#### 点数が上がる要素

- 相談の促し（「困ったら早めに相談してください」）
- チェックリスト・確認手順の作成提案
- 再発防止策（「同じミスを防ぐために〜」）
- フォロー（「様子を見る」「必要なら手伝う」）
- 一緒に改善する姿勢（「一緒に手順を決めよう」）

#### 点数が下がる要素

- 改善行動（軸3）のみで、継続的な支援・仕組みがない  
  ※軸3が高くても軸5は低くなり得る

## 3. ステータス判定（3分類）

5軸のスコアから、ゲーム上の**ステージ結果**を決定する。  
**以下の順序で判定**（上から優先）。

| 優先 | 条件 | ステータス | ゲーム表示 |
|------|------|------------|------------|
| 1 | ハラスメント度 **≥ 80** | `labor_consultation` | 労基相談エンド |
| 2 | 問題点の明確さ **< 30** または 改善行動の具体性 **< 30** | `insufficient` | 指導不足 |
| 3 | 上記以外 | `clear` | クリア |

### 判定フロー

```
即ハラスメントワード検出？
  → Yes：labor_consultation（ハラスメント度100）
  → No：続行

ハラスメント度 ≥ 80？
  → Yes：labor_consultation
  → No：続行

問題点の明確さ < 30 OR 改善行動の具体性 < 30？
  → Yes：insufficient
  → No：clear
```

### 軸4・軸5とステータスの関係

対話・確認・支援・再発防止は**ステージ成否の必須条件には含めない**。  
ただしスコアは以下に反映する。

- 結果画面の5軸表示
- ゴールドデータセットの品質評価
- 将来的な田中ステータス変化量の調整（実装予定）

| ステータス | プレイ結果 | 田中への影響（参考） |
|------------|------------|----------------------|
| `clear` | ステージクリア | 精神衛生・意識レベルが回復しやすい |
| `insufficient` | ステージ失敗（続行可） | 精神衛生・意識レベルがやや低下 |
| `labor_consultation` | ゲームオーバー | 精神衛生が大幅低下 |

## 4. アノテーション手順（ゴールドデータセット用）

### 4.1 採点の流れ

1. シナリオ（mistake / npcLine）を読む
2. 指導文（inputText）を読む
3. **即ハラスメント**に該当するか確認 → 該当ならハラスメント度100で確定
4. **5軸**をそれぞれ0〜100で採点（軸2→3→4→5→1の順がおすすめ）
5. 第3章のルールで **status** を計算（手動の「なんとなく」禁止）
6. `notes` に採点理由を1〜2文で記録

### 4.2 採点順序の推奨

```
軸2 問題点の明確さ  … 「何が問題か」わかるか
軸3 改善行動の具体性 … 「次何をするか」わかるか
軸4 対話・確認       … 聞いているか
軸5 支援・再発防止   … 仕組みがあるか
軸1 ハラスメント度   … 傷つける表現はないか
→ status 計算
```

### 4.3 複数人採点時

| 状況 | 対応 |
|------|------|
| スコア差が **20点以内** かつ status 一致 | 平均値または先着値を採用 |
| スコア差 **20点超** または status 不一致 | 議論して確定値を決める |
| 合意できない | `pattern: border` として残し、notes に争点を記録 |

### 4.4 記録する項目

| フィールド | 必須 | 説明 |
|------------|------|------|
| `harassmentScore` | ○ | 軸1：ハラスメント度（0〜100） |
| `problemClarityScore` | ○ | 軸2：問題点の明確さ（次に取るべき行動を含む）（0〜100） |
| `dialogueScore` | ○ | 軸3：対話・確認（0〜100） |
| `supportScore` | ○ | 軸4：支援・再発防止（0〜100） |
| `status` | ○ | clear / insufficient / labor_consultation |
| `matchedRiskWords` | △ | 検出したリスク表現 |
| `matchedGoodWords` | △ | 検出した建設的表現 |
| `notes` | ○ | 採点理由 |

## 5. 採点例

**シナリオ**  
- ミス：田中が朝の会議に20分遅刻した  
- 発言：すみません、電車が遅れていて……

### 例A：模範的な指導（clear）

```
指導文：
「今日の20分遅刻について確認させてください。会議開始が遅れ、
チームに迷惑がかかりました。遅刻の原因を教えてもらえますか。
次回からは会議10分前到着をルールにして、出発時間を15分早め
ましょう。チェックリストを作って、遅れそうなときは出発前に
連絡してください。困ったら早めに相談を。」

harassmentScore:         10
problemClarityScore:     82  … 遅刻・影響が明確、次の行動も具体的
dialogueScore:           75  … 原因を聞いている
supportScore:            80  … チェックリスト・相談・連絡ルール
status: clear
notes: 全軸高水準。問題特定→対話→具体行動→仕組みの流れが良い。
```

### 例B：指導不足（insufficient）

```
指導文：
「遅刻はダメだよ。次気をつけて。」

harassmentScore:         35
problemClarityScore:     25  … 「遅刻」に触れるが影響・内容・次の行動が曖昧
dialogueScore:           10  … 一方的
supportScore:            5
status: insufficient
notes: 問題点の明確さが30未満のため指導不足。
```

### 例C：労基相談エンド（labor_consultation）

```
指導文：
「お前、何回同じミスすれば気が済むんだ。社会人失格だ。辞めろ。」

harassmentScore:         95
problemClarityScore:     20
dialogueScore:           0
supportScore:            0
status: labor_consultation
notes: 人格否定＋解雇示唆。
matchedRiskWords: お前, 社会人失格, 辞めろ
```

### 例D：問題は明確だが行動は抽象的（border・clear）

```
指導文：
「20分も遅刻するのは問題です。今後は時間を守るように。」

harassmentScore:         28
problemClarityScore:     52  … 遅刻・問題指摘あり（次の行動は抽象的）
dialogueScore:           15
supportScore:            10
status: clear
notes: 問題点の明確さが30以上のためクリア。改善が抽象的でも4軸判定では通過する。
```

### 例E：行動は具体的だが対話・支援が弱い（clear）

```
指導文：
「会議に20分遅刻した件、問題です。次回から10分前到着、
出発15分早め、遅れそうなら連絡を。以上。」

harassmentScore:         15
problemClarityScore:     70  … 問題点＋具体的な次の行動
dialogueScore:           25  … 原因確認なし
supportScore:            30  … 連絡ルールのみ
status: clear
notes: 対話・支援は低いが、問題点が基準を満たすためclear。
      模範例ではないがステージは通過。
```

### 例F：境界ケース（border）

```
指導文：
「20分の遅刻は取引先への印象にも関わります。今後は会議5分前到着を
徹底してください。再発した場合は評価に反映します。」

harassmentScore:         45
problemClarityScore:     72
dialogueScore:           20
supportScore:            15
status: clear
notes: 「評価に反映」は厳しいが業務範囲内と判断。対話・支援は低め。
```

## 7. 実装との対応

| 項目 | 状態 |
|------|------|
| 本ルーブリック（4軸） | **現行基準** |
| `lib/evaluator.ts` | **4軸キーワード評価器（現行）** |
| `lib/evaluator-llm.ts` | **4軸LLM評価器（RUBRIC + Few-shot + リトライ + フォールバック）** |
| `types/game.ts` の `EvaluationResult` | **4軸（移行済み）** |
| `npm run evaluate` | キーワード評価器の精度計測 |
| `npm run evaluate:llm` | LLM評価器の精度計測（要 OPENAI_API_KEY） |

ゴールドデータセットは**本ルーブリックの4軸**で作成する。  
評価器の実装更新後、パラメータを `data/evaluation/` のデータで調整する。

### 廃止フィールド

| 旧フィールド | 扱い |
|--------------|------|
| `actionSpecificityScore`（旧軸3・改善行動の具体性） | **廃止** → `problemClarityScore` に統合 |
| `specificityScore` | 廃止 → `problemClarityScore` |
| `improvementScore` | 廃止 → `problemClarityScore` ＋ `supportScore` |
| `satisfactionScore` | 廃止 → `dialogueScore` で代替 |
